# 后台管理模板项目快速搭建指南

## 1. 项目介绍

### 做了什么

当前项目是一套基于 Vue 3 的后台管理模板，已经打通了登录、路由、状态管理、请求层、动态权限、布局系统、多主题、mock、多语言、标签页等核心能力。

### 为什么要做

新项目最耗时间的往往不是业务页面，而是后台基础设施。把这些通用能力先搭好，后续只需要往 `views`、`api`、`store` 里填业务代码即可。

### 本文档的使用方式

这份文档只负责“快速搭建”。
原有超长文档继续保留，作为完整学习记录、问题记录和排查记录使用。

## 6. 目录结构设计（说明每个目录作用）

### 做了什么

项目按“后台模板长期维护”的思路划分目录，而不是把所有文件都堆在 `src` 根目录。

### 为什么要做

目录结构清晰后，后续接真实业务、拆模块、加人协作都会轻松很多。

### 推荐目录结构

```text
src/
├─ api/                # 按业务模块拆分接口
├─ assets/             # 静态资源
├─ components/         # 通用组件
├─ constants/          # 常量定义
├─ directives/         # 自定义指令
├─ hooks/              # 组合式逻辑封装
├─ layout/             # 后台整体布局
├─ locales/            # 国际化配置与文案
├─ mock/               # 本地 mock 数据与 mock 模块
├─ permission/         # 权限相关逻辑
├─ plugins/            # 插件初始化入口
├─ request/            # Axios 实例与拦截器
├─ router/             # 路由入口与路由模块
├─ store/              # Pinia 入口与各模块 store
├─ styles/             # 全局样式、主题变量
├─ utils/              # 工具方法
└─ views/              # 页面级组件
```

### 各目录作用

| 目录             | 作用                                                           |
| ---------------- | -------------------------------------------------------------- |
| `src/api`        | 每个业务模块对应一组接口方法                                   |
| `src/components` | 可复用 UI 组件与业务模板组件                                   |
| `src/constants`  | 布局、主题、语言、存储 key、HTTP 状态码等常量                  |
| `src/directives` | 目前已接入 `v-permission` 权限指令                             |
| `src/hooks`      | 如 `useLayout`、`useTagsView`、`useLocale` 等组合式封装        |
| `src/layout`     | 左侧布局、顶部布局、混合布局，以及头部、菜单、标签页等壳子组件 |
| `src/locales`    | `vue-i18n` 实例、语言解析、语言包                              |
| `src/mock`       | mock 开关、延迟模拟、账号数据、权限数据、Dashboard 数据        |
| `src/permission` | 动态路由注册、路由守卫、按钮权限判断                           |
| `src/plugins`    | 项目启动时统一安装 Pinia、i18n、路由守卫、指令                 |
| `src/request`    | Axios 实例、请求拦截、错误处理                                 |
| `src/router`     | 静态路由、动态路由、路由辅助方法                               |
| `src/store`      | App、User、Permission、Tabs 等核心状态                         |
| `src/styles`     | 全局变量、主题、Element Plus 变量映射、响应式样式              |
| `src/views`      | 登录页、首页、系统管理、错误页等页面                           |

### 关键结论

目录一定要服务于“分层”和“复用”，不要图省事把页面、接口、状态混在一起。

## 7. 路由搭建

### 做了什么

当前路由体系包含：

- 静态路由：登录页、首页、403、404、500
- 动态路由：系统管理模块
- 多语言前缀路由：如 `/zh-CN/dashboard`、`/en-GB/system/user`
- 路由辅助层：用于路径拼接、菜单转换、语言前缀处理

### 为什么要做

后台项目几乎都会有权限路由、隐藏页、详情页、编辑页、404/403 页面，所以路由必须从一开始就设计成“可扩展结构”。

### 关键文件

- `src/router/index.js`
- `src/router/modules/static.js`
- `src/router/modules/async.js`
- `src/router/modules/system.js`
- `src/router/helper.js`

### 关键配置

`src/router/index.js`

```js
const router = createRouter({
  history: createWebHistory(import.meta.env.BASE_URL),
  routes: staticRoutes,
  scrollBehavior() {
    return { left: 0, top: 0 }
  }
})
```

动态路由示例：

```js
export const systemRoute = {
  path: `/${LOCALE_ROUTE_PARAM}/system`,
  component: Layout,
  meta: {
    title: '系统管理',
    requiresAuth: true,
    permissionCodes: ['system:access']
  }
}
```

### 推荐的路由 meta 字段

| 字段                 | 用途                             |
| -------------------- | -------------------------------- |
| `title` / `titleKey` | 菜单标题、面包屑标题、标签页标题 |
| `icon`               | 菜单图标                         |
| `requiresAuth`       | 是否必须登录                     |
| `permissionCodes`    | 页面级权限码                     |
| `hidden`             | 是否完全隐藏                     |
| `hideMenu`           | 是否不显示在菜单中               |
| `activeMenu`         | 详情/编辑页高亮哪个菜单          |
| `affix`              | 是否固定标签页                   |
| `keepAlive`          | 是否需要缓存                     |

### 关键结论

路由不要只当“页面跳转表”来写，它还会驱动菜单、面包屑、标签页、权限判断和页面缓存。

## 8. 状态管理（Pinia）

### 做了什么

项目把核心状态拆成 4 个 store：

- `app`：主题、语言、布局模式、设备类型、菜单展开状态
- `user`：token、用户信息、登录状态
- `permission`：页面权限码、按钮权限码、菜单、动态路由状态
- `tabs`：访问过的标签页、缓存页面列表

### 为什么要做

后台项目的“全局状态”比较多，如果不拆模块，后面会很乱。

### 关键文件

- `src/store/index.js`
- `src/store/modules/app.js`
- `src/store/modules/user.js`
- `src/store/modules/permission.js`
- `src/store/modules/tabs.js`

### 关键配置

`src/store/index.js`

```js
export function initializeStore() {
  const userStore = useUserStore(pinia)
  const appStore = useAppStore(pinia)
  const permissionStore = usePermissionStore(pinia)
  const tabsStore = useTabsStore(pinia)

  userStore.restoreUserState()
  appStore.restoreAppState()
  permissionStore.restorePermissionState()
  tabsStore.restoreTabsState()
}
```

本地持久化 key：

```js
export const APP_STORAGE_KEY = 'ai-v3-app-store'
export const USER_STORAGE_KEY = 'ai-v3-user-store'
export const PERMISSION_STORAGE_KEY = 'ai-v3-permission-store'
export const TABS_STORAGE_KEY = 'ai-v3-tabs-store'
```

### 关键结论

Pinia 不只是存数据，还要负责“恢复状态”和“同步状态到本地缓存”。

## 9. 请求封装（Axios）

### 做了什么

项目基于 Axios 创建了统一请求实例，封装了：

- `baseURL` 与超时控制
- token 自动注入
- 业务成功码判断
- 401 / 403 / 500 统一处理
- `get/post/put/delete` 二次导出

### 为什么要做

如果每个页面都手写请求、手写报错提示、手写 token 逻辑，后面维护成本会非常高。

### 关键文件

- `src/request/index.js`
- `src/request/helper.js`
- `src/api/*`

### 关键配置

```js
const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || '/api',
  timeout: 10000
})
```

```js
if (accessToken && nextConfig.headers.isToken !== false) {
  nextConfig.headers.Authorization = `Bearer ${accessToken}`
}
```

```js
if (REQUEST_SUCCESS_CODE_LIST.includes(businessCode)) {
  return responseData.data !== undefined ? responseData.data : responseData
}
```

### API 层组织方式

```js
export function loginApi(loginForm = {}) {
  if (isMockEnabled()) {
    return createMockRequest(() => mockLogin(loginForm), 800)
  }

  return request.post('/auth/login', loginForm, {
    headers: { isToken: false }
  })
}
```

### 关键结论

页面层只管“调用接口和渲染结果”，不要把 token、错误码、跳转逻辑散落到各个页面里。

## 10. 权限设计（动态路由 + 按钮权限）

### 做了什么

项目把权限拆成两层：

- 页面级权限：控制动态路由和菜单
- 按钮级权限：控制按钮是否显示

### 为什么要做

后台项目里，“能不能进页面”和“进页面后能不能点按钮”不是一回事，必须分开设计。

### 关键文件

- `src/permission/index.js`
- `src/permission/guard.js`
- `src/permission/utils.js`
- `src/directives/permission.js`
- `src/store/modules/permission.js`

### 登录后的权限链路

1. 调 `loginAction()` 获取 token。
2. 调 `fetchUserInfoAction()` 获取用户信息和页面权限码。
3. 调菜单接口、按钮权限接口拿到菜单数据和按钮码。
4. 调 `permissionStore.buildPermissionState()` 生成权限状态。
5. 调 `rebuildPermissionRoutes()` 重新注入动态路由。

### 关键配置

动态路由注册：

```js
export function setupPermissionRoutes() {
  const accessibleRouteList = permissionStore.generatePermissionState()

  if (accessibleRouteList.length > 0) {
    addDynamicRoutes(accessibleRouteList)
  }

  permissionStore.setDynamicRouteAdded(true)
}
```

路由守卫：

```js
if (!hasToken) {
  return buildLoginRedirectLocation(to, routeLocale)
}

if (!permissionStore.dynamicRouteAdded) {
  setupPermissionRoutes()
  return to.fullPath
}
```

按钮权限指令：

```vue
<el-button v-permission="'system:user:create'">新增用户</el-button>
```

### 关键结论

页面权限靠 `meta.permissionCodes`，按钮权限靠 `buttonPermissionList + v-permission`，不要混用。

## 11. 布局设计（Layout）

### 做了什么

项目提供 3 种后台布局模式：

### 为什么要做

布局系统是后台模板的壳子，壳子不稳，后续所有业务页面都会受影响。

### 关键文件

- `src/layout/index.vue`
- `src/layout/modes/LeftLayout.vue`
- `src/layout/modes/TopLayout.vue`
- `src/layout/modes/MixLayout.vue`
- `src/layout/components/*`
- `src/hooks/useLayout.js`
- `src/hooks/useLayoutMenu.js`

### 关键配置

```js
const currentLayoutComponent = computed(() => {
  const layoutComponentMap = {
    left: LeftLayout,
    top: TopLayout,
    mix: MixLayout
  }

  return layoutComponentMap[layoutMode.value] || LeftLayout
})
```

### 已经具备的布局能力

- 响应式桌面端 / 移动端切换
- 侧边栏折叠
- 顶部菜单与混合布局菜单联动
- 面包屑
- 标签页导航
- 退出登录、主题切换、语言切换、布局切换
- 菜单展开状态持久化

### 关键结论

布局模式切换不要散在组件里判断，统一交给 `layout/index.vue` 做总入口最稳。

## 12. 主题与样式方案（CSS Variables / 多主题）

### 做了什么

项目采用“设计令牌 + 主题变量 + Element Plus 映射”的样式结构：

- `variables.less`：不随主题变化的尺寸、间距、圆角、阴影
- `themes/default.less`：默认主题
- `themes/dark.less`：暗黑主题
- `themes/deep-blue.less`：深蓝主题
- `element.less`：把项目变量映射给 Element Plus
- `utils/theme.js`：把主题挂到 DOM

### 为什么要做

多主题场景下，直接在组件里写死颜色很难维护。用 CSS Variables 可以把主题切换成本降到最低。

### 关键文件

- `src/styles/index.less`
- `src/styles/variables.less`
- `src/styles/themes/*.less`
- `src/utils/theme.js`
- `src/store/modules/app.js`

### 关键配置

```less
:root {
  --app-layout-header-height: 56px;
  --app-layout-sidebar-width: 220px;
  --app-shadow-base: 0 6px 18px rgba(15, 23, 42, 0.1);
}
```

```less
html[data-theme='default'] {
  --app-color-primary: #409eff;
  --app-color-bg-page: #f5f7fa;
  --app-color-text-primary: #303133;
}
```

```js
html.setAttribute('data-theme', nextTheme)
html.style.colorScheme = DARK_THEME_LIST.includes(nextTheme) ? 'dark' : 'light'
```

### 关键结论

尺寸和颜色一定分层管理：尺寸放设计令牌，颜色放主题文件，组件只消费变量。

## 13. mock 数据方案

### 做了什么

项目内置了一套轻量 mock 方案，不依赖独立 mock 服务，直接通过环境变量控制是否启用。

### 为什么要做

后台项目早期经常是“前端先跑页面，后端接口还没完全就绪”。本地 mock 可以先把登录、权限、菜单、首页这些链路跑通。

### 关键文件

- `src/mock/helper.js`
- `src/mock/modules/auth.js`
- `src/mock/modules/permission.js`
- `src/mock/modules/dashboard.js`
- `src/mock/data/*`
- `.env.development`

### 关键配置

```js
export function isMockEnabled() {
  return import.meta.env.VITE_USE_MOCK === 'true'
}
```

```js
export function createMockRequest(executor, delay = 500) {
  return new Promise((resolve, reject) => {
    window.setTimeout(() => {
      try {
        resolve(cloneMockData(executor()))
      } catch (error) {
        reject(error)
      }
    }, delay)
  })
}
```

### 当前 mock 覆盖内容

- 登录
- 获取用户信息
- 菜单权限
- 按钮权限
- Dashboard 数据
- 系统管理演示数据

### 关键结论

mock 开关要放在 API 层统一处理，不要在页面里到处写 if/else。

## 15. 项目启动与构建方式

### 做了什么

项目已经把本地开发、预发构建、生产构建命令都整理好了。

### 为什么要做

后台项目通常至少会有开发、预发、生产 3 套环境，脚本越清晰越不容易出错。

### 常用命令

```bash
pnpm install
pnpm dev
pnpm dev:staging
pnpm build
pnpm build:staging
pnpm build:prod
pnpm preview
pnpm lint
pnpm format
```

### 命令说明

| 命令                 | 作用               |
| -------------------- | ------------------ |
| `pnpm dev`           | 启动开发环境       |
| `pnpm dev:staging`   | 用预发环境变量启动 |
| `pnpm build`         | 默认构建           |
| `pnpm build:staging` | 预发构建           |
| `pnpm build:prod`    | 生产构建           |

- 是否启用 mock：`VITE_USE_MOCK`

## 16. 常见问题总结（只保留必要结论）

### 1. 登录成功后没有进入目标页面

先检查是否执行了“获取用户信息 -> 获取菜单/按钮权限 -> `buildPermissionState()` -> `rebuildPermissionRoutes()`”这条链路。

### 2. 页面刷新后菜单或权限丢失

确认项目启动时已经执行 `initializeStore()`，并且在有 token 时执行了 `setupPermissionRoutes()`。

### 3. 切到真实后端后还在走本地 mock

确认对应环境文件里的 `VITE_USE_MOCK=false`。

### 4. 动态菜单不显示或 404

先检查路由 `meta.permissionCodes` 和用户权限码是否匹配，再检查动态路由是否已经注入。

### 5. 按钮权限不生效

确认按钮权限码已经写入 `buttonPermissionList`，并且页面使用了 `v-permission` 指令。

### 6. 主题或语言切换后刷新失效

确认 `app store` 已持久化状态，并且启动时执行了 `restoreAppState()`。

### 7. 构建时报环境相关的 `spawn EPERM`

如果是在受限的 Windows 沙箱环境里运行构建，这通常是环境限制，不一定是项目代码问题；优先在正常终端环境重新执行一次 `pnpm build`。

## 最后建议

这套模板的推荐搭建顺序是：

1. 先起项目骨架和目录。
2. 再接工程化配置。
3. 再接路由、Pinia、请求层。
4. 然后打通登录、权限、布局、主题、mock。
5. 最后再往里面填业务模块。

按这个顺序做，项目会一直保持“当前可运行、后续可扩展”的状态。
