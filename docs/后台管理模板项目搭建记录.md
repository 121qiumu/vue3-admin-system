## 4. 第 1 步：初始化项目骨架

### 4.1 这一步的目标

先把 Vue 3 项目真正创建出来，并把后台管理系统常用目录提前规划好。

这一步不写业务逻辑，只做两件事：

1. 创建一个最小可运行的 Vue 3 + Vite + JavaScript 项目
2. 把后台项目后续要用到的基础目录先搭出来

### 4.2 这一步的核心

推荐命令：

```bash
cd /d/GitHub-Project/vue3-admin-system
pnpm create vite . --template vue
pnpm install
pnpm dev
```

### 4.3 为什么这样做

原因很简单：

1. 用官方脚手架起项目，稳定、主流、学习资料也多
2. 先搭目录，再做业务，后面扩展起来更顺
3. 后台项目比普通展示页更依赖清晰分层，所以目录设计要早点定下来

### 4.4 第 1 步的重点

这一步最重要的不是“写代码”，而是“搭结构”。

必须先让下面这些目录有明确位置：

1. `src/api`
2. `src/components`
3. `src/layout`
4. `src/router`
5. `src/store`
6. `src/views`
7. `src/styles`
8. `src/utils`
9. `src/hooks`
10. `src/plugins`
11. `src/directives`
12. `src/permission`
13. `src/mock`
14. `src/constants`

### 4.5 第 1 步的难点

对初学者来说，第 1 步最容易出现两个问题：

1. 一开始就想写业务页面，导致后面目录越来越乱
2. 把所有文件都堆在 `src` 根目录，后续维护会非常痛苦

所以第 1 步最重要的是建立“分层意识”。

## 5. 第 2 步：安装依赖与工程化配置

### 5.1 这一步的目标

把项目真正开发后台项目所需要的核心依赖都接进来，并建立统一的工程规范。

### 5.2 这一步安装了什么

#### 生产依赖

```bash
pnpm add element-plus vue-router pinia axios
```

#### 开发依赖

```bash

```

### 5.3 第 2 步的核心

这一阶段主要完成了下面几件事：

1. 接入 Element Plus
2. 接入 Vue Router
3. 接入 Pinia
4. 接入 Axios
5. 接入 Less
6. 接入 ESLint
7. 接入 Prettier
8. 接入 Husky
9. 接入 lint-staged

### 5.4 为什么这样做

后台项目开发时间通常比较长、参与人比较多，所以越早把规范定下来越好。

如果一开始不做这些基础配置，后面常见问题会越来越多：

1. 代码风格混乱
2. 导包方式不统一
3. 提交前没人检查代码质量
4. 请求层没有统一入口
5. 状态管理和路由入口不清楚

### 5.5 第 2 步的重点

这一步最重要的是理解“工程化不是多余的配置，而是长期维护的基础”。

当前项目已经完成的核心点包括：

1. `main.js` 已经通过 `plugins` 目录统一注册插件
2. `router` 已经具备最小可运行能力
3. `store` 已经具备 Pinia 入口
4. `utils/request.js` 已经具备 Axios 实例入口
5. `eslint.config.js` 和 `prettier.config.cjs` 已经形成分工协作
6. `.husky/pre-commit` 已经接入 `pnpm lint-staged`

### 5.6 第 2 步的难点

这一步的难点通常有三个：

1. 不知道哪些依赖是运行时需要，哪些是开发时需要
2. 不知道 ESLint 和 Prettier 应该怎么分工
3. 不知道 Husky 和 lint-staged 为什么要一起用

当前项目采用的思路是：

1. 运行时真正会用到的放 `dependencies`
2. 构建、规范、格式化、自动导入这类工具放 `devDependencies`
3. ESLint 管“代码质量”
4. Prettier 管“代码格式”
5. Husky 负责“提交前执行”
6. lint-staged 负责“只检查暂存区文件”

## 6. 第 3 步：Vite 配置

### 6.1 这一步的目标

让 Vite 真正具备后台项目常用能力，同时保持配置尽量简单。

这一步主要完成下面这些内容：

1. 配置 `@` 路径别名
2. 配置 Element Plus 自动导入
3. 配置组件自动导入
4. 配置图标自动导入
5. 配置开发服务器参数
6. 预留代理配置位置
7. 配置 Less 预处理器

### 6.2 第 3 步新增的重点依赖

```bash
pnpm add -D unplugin-icons @iconify-json/ep
```

### 6.3 为什么要这样配置

#### 1. `@` 路径别名

如果没有别名，项目一大就会出现大量这种路径：

```js
../../../../utils/request
```

这类路径非常不利于维护，所以必须尽早统一成：

```js
@/utils/request
```

#### 2. Element Plus 自动导入

Element Plus 组件很多，如果每个页面都手动 import：

1. 重复劳动多
2. 文件会越来越长
3. 初学者容易漏掉样式和组件引入

所以这里采用官方主流推荐思路，用插件自动按需导入。

#### 3. 组件自动导入

组件自动导入不仅能处理 Element Plus，也能为后续统一管理图标组件打基础。

#### 4. 图标自动导入

后台项目里图标非常多。

如果每次都手动引入图标组件，会有两个问题：

1. 写法繁琐
2. 每个页面风格不统一

当前选择的是统一图标组件方案：

1. 使用 `unplugin-icons`
2. 当前先启用 `ep` 图标集
3. 组件写法统一成 `<IconEpHomeFilled />`

这样后续非常适合后台项目扩展。

#### 5. 开发服务器与代理

后台项目前后端联调时，最常见的问题就是端口、主机、跨域代理。

所以这一步先把下面这些位置预留出来：

1. 主机地址
2. 端口
3. 是否自动打开浏览器
4. 接口前缀
5. 代理目标地址

#### 6. Less 预处理器

虽然项目后面会以 CSS Variables 来处理主题切换，但 Less 仍然适合处理下面这些“编译阶段就能确定”的变量：

1. 布局尺寸
2. 边距
3. 圆角
4. 一些基础结构样式

### 6.4 第 3 步的重点

这一步最重要的不是记住插件名字，而是理解 Vite 配置是“项目能力入口”。

当前项目的 Vite 配置已经实现：

1. Vue 基础编译
2. `@` 路径别名
3. Element Plus 自动导入
4. 组件自动导入
5. 图标自动导入
6. 开发服务器配置
7. 代理预留
8. Less 预处理能力

### 6.5 第 3 步的难点

第 3 步对初学者来说最容易混淆的点有：

1. `AutoImport` 和 `Components` 有什么区别
2. 图标自动导入到底是“API 自动导入”还是“组件自动导入”
3. 代理配置为什么不能直接写死

这里统一解释一下：

1. `AutoImport` 负责自动导入函数和 API，例如 `ref`、`computed`、`useRouter`
2. `Components` 负责自动导入组件，例如 `ElButton`、`IconEpHomeFilled`
3. 图标在当前方案里本质上是“组件”，所以由 `Components` 处理更合理
4. 代理目标通常跟环境有关，所以不要在配置里写死，应该预留给环境变量

## 11. 第 4 步：全局样式与主题系统

### 11.1 这一步的目标

这一阶段要解决一个后台项目里非常常见的问题：

1. 全局样式放在哪里
2. 主题颜色怎么统一管理
3. 主题切换时怎么尽量少改代码
4. 如何让自定义页面样式和 Element Plus 组件风格保持一致

### 11.2 这一步完成了什么

当前已经完成下面这些能力：

1. 建立 `styles` 目录规范
2. 使用 CSS Variables 管理颜色、间距、圆角、布局尺寸
3. 支持三套主题：`default`、`dark`、`deep-blue`
4. 使用 `data-theme` 挂载主题到 `html` 和 `body`
5. 使用 `localStorage` 持久化当前主题
6. 建立 Element Plus 变量适配层
7. 在 Dashboard 页面加入了主题切换演示

### 11.3 styles 目录建议

当前推荐目录结构如下：

```text
src/styles/
├─ index.less
├─ reset.less
├─ variables.less
├─ element.less
└─ themes/
   ├─ default.less
   ├─ dark.less
   └─ deep-blue.less
```

它们的职责分别是：

1. `index.less`：统一样式入口，只负责组织加载顺序
2. `reset.less`：浏览器默认样式重置与基础全局样式
3. `variables.less`：主题无关的全局设计令牌，例如间距、圆角、布局尺寸
4. `element.less`：把项目主题变量映射给 Element Plus
5. `themes/*.less`：每一套主题自己的颜色变量

### 11.4 为什么这样分层

这样分层的好处非常明显：

1. 不同职责的样式不会混在一起
2. 切换主题时主要改颜色，不需要到处找组件样式
3. 以后新增第四套、第五套主题时，只需要继续新增主题文件
4. 与 Element Plus 的关系更清楚，不会出现“项目自己的变量”和“组件库变量”互相打架

### 11.5 这一步的重点

这一阶段最重要的是理解：

“主题系统的核心不是切换按钮，而是变量设计。”

如果变量命名不清楚，后面主题一多就会越来越难维护。

当前项目把变量拆成了两层：

1. 主题无关变量：间距、尺寸、圆角、阴影
2. 主题相关变量：主色、背景色、文本色、边框色、侧边栏色、头部色

### 11.6 这一步的难点

第 4 步最容易踩坑的地方有：

1. 把所有颜色和尺寸都写死在组件里
2. 一边用 Less 变量，一边又零散写颜色，导致风格难统一
3. 切换主题时只改页面背景，没有同步 Element Plus 组件变量
4. 主题切换后没有持久化，刷新页面又回到默认主题

当前项目采用的解决方式是：

1. 优先使用 CSS Variables 管理设计令牌
2. 使用 `html[data-theme='xxx']` 作为主题入口
3. 用单独的 `element.less` 适配 Element Plus 变量
4. 用 `localStorage` 缓存当前主题
5. 项目启动时先执行主题初始化，减少首次闪烁

## 12. 第 5 步：路由系统基础搭建

### 12.1 这一步的目标

这一阶段的核心任务是把后台项目最基本的路由骨架搭起来，并且提前为“权限路由”和“动态菜单”留好扩展空间。

目标包括：

1. 接入 Vue Router
2. 区分静态路由与动态路由
3. 提供基础 Layout 容器结构
4. 让登录页、首页、403、404 全部跑通
5. 为后续权限系统预留动态路由注入能力

### 12.2 当前 router 目录结构建议

```text
src/router/
├─ index.js
└─ modules/
   ├─ static.js
   ├─ async.js
   └─ system.js
```

每个文件的职责如下：

1. `index.js`：创建路由实例，并提供动态路由的添加与重置方法
2. `modules/static.js`：放项目启动就能直接注册的静态路由
3. `modules/async.js`：统一导出动态路由列表
4. `modules/system.js`：一个具体的动态路由业务模块示例

### 12.3 为什么要拆成静态路由和动态路由

这是后台项目里非常常见的一种设计。

原因是：

1. 登录页、404 这类页面，任何时候都应该可访问，所以适合放静态路由
2. 用户管理、角色管理、菜单管理这类页面，通常跟权限有关，所以适合放动态路由
3. 这样后续做权限控制时，思路会非常清楚：
   - 先注册静态路由
   - 登录后再根据权限动态注册业务路由

### 12.4 这一步的重点

这一阶段最重要的是建立两层结构：

1. 根级路由入口
2. Layout 容器路由

当前项目使用的是典型后台结构：

1. `App.vue` 只负责根级 `router-view`
2. 真正的后台框架放在 `layout/index.vue`
3. 首页等业务页面作为 Layout 的 children 子路由来渲染

这类结构有一个非常大的好处：

后续无论你要加侧边栏、面包屑、标签页还是权限菜单，都会有明确的位置可以放。

### 12.5 路由懒加载方案

当前项目采用的是页面级懒加载：

```js
component: () => import('@/views/dashboard/index.vue')
```

好处是：

1. 首屏加载更轻
2. 每个页面会单独拆分 chunk
3. 后台项目页面通常很多，这种方式几乎是默认推荐方案

当前 Layout 容器采用的是直接导入，因为它属于高频基础框架，保持直观更适合学习阶段。

### 12.6 这一步的难点

初学者在这一步最容易混淆的地方有：

1. 为什么首页不是直接写在 `App.vue`
2. 为什么还没做权限，就先要准备动态路由结构
3. 为什么要给路由写 `meta`

这里统一解释：

1. `App.vue` 越简单越好，后台框架应该交给路由去决定
2. 动态路由结构早点定下来，后面做权限时就不需要大改目录
3. `meta` 会在后续被菜单、面包屑、标签页、权限判断复用，所以现在就应该开始规范

## 14. 第 7 步：Pinia 状态管理设计

### 14.1 这一步的目标

这一阶段的核心不是“能不能用 store”，而是“怎么把 store 设计得清楚、稳定、好扩展”。

后台项目里的全局状态通常很多，如果一开始全堆进一个 store，后面会越来越乱。

所以这一步的目标是：

1. 使用 Pinia
2. 用模块化方式拆分 store
3. 为用户、权限、应用设置、标签页分别建立独立 store
4. 提前考虑刷新后状态恢复

### 14.2 当前 store 目录结构建议

```text
src/store/
├─ index.js
└─ modules/
   ├─ app.js
   ├─ permission.js
   ├─ tabs.js
   └─ user.js
```

### 14.3 每个 store 负责什么

#### 1. user store

负责用户会话相关状态，例如：

1. token
2. 用户信息
3. 登录状态
4. 展示名称

这一类状态的特点是：

1. 强依赖登录流程
2. 刷新后通常还需要保留
3. 退出登录时需要统一清空

#### 2. permission store

负责权限与菜单相关状态，例如：

1. 权限码列表
2. 可访问动态路由列表
3. 当前菜单列表
4. 动态路由是否已注入

这一类状态的特点是：

1. 与用户身份强相关
2. 会影响菜单、路由、按钮权限
3. 路由记录本身不适合直接持久化，需要通过权限码重新生成

#### 3. app store

负责应用级设置状态，例如：

1. 当前主题
2. 当前布局模式
3. 侧边栏折叠状态
4. 全屏状态
5. 设备类型
6. 移动端侧边栏显示状态

这一类状态的特点是：

1. 跟具体业务无关
2. 整个后台壳子都会依赖它
3. 通常适合做本地持久化

#### 4. tabs store

负责标签页导航相关状态，例如：

1. 已访问标签页列表
2. 需要缓存的页面名称列表

这一类状态的特点是：

1. 与路由联动明显
2. 刷新后如果能恢复，用户体验会更好
3. 后续右键菜单、关闭其它、关闭全部等能力都建立在它之上

### 14.4 为什么这样拆分

这样拆分的原因很重要：

1. user 只管“我是谁、是否登录”
2. permission 只管“我能看什么、能访问什么”
3. app 只管“后台壳子长什么样”
4. tabs 只管“当前打开了哪些页面”

每个模块职责单一后，会有几个明显好处：

1. 字段更容易命名
2. action 更容易写清楚
3. 后续找问题时更容易定位
4. 退出登录、切换账号、刷新恢复时也更好处理

### 14.5 示例状态设计

当前项目中的状态设计思路如下：

#### user

1. `token`
2. `userInfo`
3. `isLogin`
4. `displayName`

#### permission

1. `permissionCodeList`
2. `accessibleRouteList`
3. `menuRouteList`
4. `dynamicRouteAdded`

#### app

1. `theme`
2. `layoutMode`
3. `sidebarCollapsed`
4. `fullscreenActive`
5. `deviceType`
6. `mobileSidebarVisible`

#### tabs

1. `visitedViewList`
2. `cachedViewNameList`

### 14.6 为什么所有 store 都使用组合式写法

当前项目统一使用的是 Pinia 的 setup store 写法。

这样做的好处是：

1. 更接近 Vue 3 组合式 API 的思维方式
2. `ref`、`computed`、方法都能自然组合在一起
3. 后续抽复用逻辑时更灵活
4. 对已经习惯组合式 API 的同学更友好

### 14.7 如何处理刷新后状态恢复

这一阶段没有引入额外的持久化插件，而是采用了“手动持久化 + 初始化恢复”的方式。

当前项目做法是：

1. 新增 `src/utils/storage.js`
2. 新增 `src/constants/storage.js`
3. 每个 store 自己决定哪些状态需要持久化
4. 在 `src/store/index.js` 中提供统一的 `initializeStore()`
5. 项目启动时，先安装 Pinia，再统一恢复各 store 的缓存状态

### 14.8 哪些状态适合持久化，哪些不适合

适合持久化的：

1. token
2. 用户基础信息
3. 当前主题
4. 当前布局模式
5. 侧边栏折叠状态
6. 已访问标签页列表
7. 权限码列表

不建议直接持久化的：

1. 路由记录本身
2. 组件实例状态
3. 纯运行时临时状态
4. 由窗口宽度实时推导出来的设备类型

比如权限模块中，`accessibleRouteList` 不直接持久化，而是刷新后先恢复 `permissionCodeList`，再根据本地路由配置重新生成。

这是一种更稳、更适合后台项目的做法。

### 14.9 当前已经完成的状态迁移

这一步不仅新增了 store，还把前面已有的几类全局状态迁移进了 Pinia：

1. 主题状态从原先的共享 hook 迁移到了 `app store`
2. 布局状态从原先的共享 hook 迁移到了 `app store`
3. 标签页状态从原先的共享 hook 迁移到了 `tabs store`
4. Layout 菜单开始改为读取 `permission store` 中的菜单数据

这样后续权限系统和登录系统接起来时，就会顺很多。

## 15. 第 8 步：登录页与登录流程

### 15.1 这一步的目标

这一阶段要把后台项目里最常见的一条主链路先跑通：

1. 用户输入用户名和密码
2. 表单校验通过后发起登录
3. 登录成功后保存 token
4. 再拉取当前用户信息
5. 根据用户信息生成权限与菜单
6. 最后跳转到后台首页

这一步先使用 mock 接口，不接真实后端。

### 15.2 这一步新增了什么

当前新增或重点接入了下面这些文件：

1. `src/mock/modules/auth.js`
   负责模拟登录、获取用户信息、退出登录、演示账号数据。

2. `src/api/auth.js`
   负责把 mock 数据包装成更接近真实接口请求的调用方式。

3. `src/views/login/index.vue`
   负责登录页界面、表单校验、按钮 loading、错误提示、成功跳转。

4. `src/store/modules/user.js`
   负责登录、保存 token、拉取用户信息、退出登录。

5. `src/store/modules/permission.js`
   负责保存权限码，并基于权限码生成动态菜单与动态路由数据。

6. `src/permission/index.js`
   负责统一处理权限路由的初始化、重建、清空。

7. `src/plugins/index.js`
   负责项目启动时恢复本地缓存中的登录态和权限路由。

### 15.3 这一步的核心流程

当前登录流程可以理解成下面这张“文字流程图”：

1. 用户进入登录页
2. 输入用户名和密码
3. Element Plus 表单执行校验
4. 调用登录接口 `loginApi`
5. 登录接口返回 token
6. token 保存到 `user store` 和 `localStorage`
7. 调用获取用户信息接口 `getUserInfoApi`
8. 把用户信息保存到 `user store`
9. 从用户信息里读取 `permissionCodeList`
10. 把权限码交给 `permission store`
11. 根据权限码过滤动态路由并生成菜单
12. 重新注入当前账号可访问的动态路由
13. 跳转到 `/dashboard` 或登录前原始目标页

### 15.4 为什么这一步要这样做

很多初学者会在登录接口返回成功后，直接认为“登录完成了”。

但在真实后台项目里，通常还差至少两步：

1. 拉取当前用户信息
2. 根据用户信息生成权限、菜单和动态路由

如果少了这两步，页面虽然看起来“登录成功”了，但系统并不知道：

1. 当前用户是谁
2. 当前用户能访问哪些页面
3. 左侧菜单应该显示什么

所以这一步的关键不是“点一下登录按钮”，而是把登录后的用户上下文真正建立起来。

### 15.5 这一步的重点

1. 登录页使用 Element Plus 表单，保证学习者能看到后台项目最常见的写法。
2. 登录按钮有 loading 状态，避免重复提交。
3. 登录失败会给出明显提示，不会静默失败。
4. token 会持久化保存，刷新页面后仍能恢复登录态。
5. 动态路由不会直接写死，而是根据权限码重新生成。
6. 登录页额外展示了 mock 账号卡片，方便切换不同角色体验权限差异。

### 15.6 这一步的难点

第 8 步最容易忽略的几个问题是：

1. 登录成功了，但没有保存 token，刷新页面就丢失状态
2. 保存了 token，但没有拉用户信息，页面不知道当前用户是谁
3. 拉了用户信息，但没有生成权限菜单，登录后依然看不到正确页面
4. 切换账号后没有清理旧标签页和旧动态路由，导致界面状态串号

当前项目已经把这些问题都提前处理掉了。

2. `editor / 123456`
   这个账号权限更少，更适合用来观察“不同账号看到的菜单不一样”。

## 16. 第 9 步：Axios 请求封装

### 16.1 进入这一步前先修复了什么问题

在进入请求封装前，先补上了后台项目非常关键的一层：路由守卫。

之前项目虽然已经有登录页、token 和权限菜单生成逻辑，但还没有真正把“登录状态检查”挂到路由切换过程中，所以会出现一个明显问题：

1. 没登录也能直接访问 `/dashboard`
2. 路由不会自动拦截并跳到 `/login`

这次已经补上统一路由守卫，当前可以做到：

1. `index.js` 只管 axios 实例和拦截器主流程
2. `helper.js` 只管细节辅助逻辑
3. `constants/http.js` 只管状态码和错误提示常量
4. `api/*.js` 只管具体业务接口

### 16.7 token 注入逻辑说明

token 注入发生在请求拦截器里：

1. 先从 `user store` 读取当前 token
2. 如果 token 存在，就自动拼到 `Authorization` 头中
3. 默认写法为：

```http
Authorization: Bearer xxxxxx
```

如果某个接口明确不需要 token，可以在业务层请求里传：

```js
headers: {
  isToken: false
}
```

### 16.8 错误处理逻辑说明

当前错误处理分成两层：

#### 1. 业务码错误

如果后端响应体里有 `code` 字段，就会先按业务码判断：

1. `0 / 200` 视为成功
2. `401` 视为登录失效
3. `403` 视为无权限
4. 其它业务码统一弹出错误提示

#### 2. HTTP 错误

如果请求直接在 HTTP 层失败，则按状态码处理：

1. `401` 自动清理登录态并跳登录页
2. `403` 跳到 403 页面
3. 其它状态码按统一错误提示显示
4. 如果连 response 都没有，一般视为网络异常或本地服务未启动

### 16.9 业务接口模块示例说明

当前新增了 `src/api/user.js` 作为示例，演示了后台项目里最常见的一组用户管理接口：

1. 获取用户列表

4. 更新用户
5. 删除用户

这一层的核心思想是：

这样页面层就能专注处理交互和展示，而不是请求细节。

### 16.10 这一步的难点

1. 什么都不封装，导致每个页面都自己写 token、错误提示、状态码判断
2. 过度封装，写出一堆难以理解的适配器和中间层，初学者完全看不懂

当前项目选择的是“刚好够用”的方案：

1. 保留 axios 原本的直观使用方式
2. 把公共能力统一放到拦截器
3. 只拆最有必要的辅助文件
4. 不引入复杂请求工厂或泛型封装

### 18.3 页面级权限是怎么做的

当前页面级权限的核心思路是：

1. 在动态路由的 `meta.permissionCodes` 中声明页面访问所需的权限码
2. 登录后拿到当前用户的页面级权限码列表
3. 用 `permission store` 过滤 `asyncRoutes`
4. 生成当前账号真正可访问的动态路由
5. 统一通过 `router.addRoute` 动态注入

这意味着：

1. `admin` 可以进入 `/system/user`
2. `editor` 因为没有对应页面权限码，所以不会生成这条动态路由
3. 如果强行输入地址，最终会被统一带到 403 或登录页

### 18.4 按钮级权限是怎么做的

按钮级权限的数据来源是 `permission store` 中的 `buttonPermissionList`。

当前实现了两种使用方式：

#### 1. 权限指令

全局注册了 `v-permission`。

支持的写法有：

```vue
<el-button v-permission="'system:user:create'">新增</el-button>
<el-button v-permission="['a', 'b']">需要同时满足多个权限</el-button>
<el-button v-permission="{ mode: 'any', codes: ['a', 'b'] }">满足任意一个即可</el-button>
```

#### 2. 权限函数

在 `src/permission/utils.js` 中提供了：

1. `hasPermission`
2. `hasAnyPermission`
3. `checkPermission`

这样在 `script setup` 中就可以很自然地写：

```js
const canShowExportAction = computed(() => {
  return hasPermission('system:user:export')
})
```

### 18.5 路由守卫当前做了什么

当前守卫已经完整处理下面这些场景：

1. 未登录访问受保护页面时，跳到 `/login`
2. 登录后再次访问 `/login` 时，自动跳首页或原始目标页

### 18.6 动态路由刷新后为什么还能恢复

这个问题在后台项目里非常常见。

当前项目用了两层保障：

#### 第一层：应用启动阶段提前恢复

在 `plugins/index.js` 中：

1. 先恢复 Pinia 缓存状态
2. 如果本地还保留 token，就提前调用 `setupPermissionRoutes()`
3. 让动态路由在 Router 首次匹配前就先挂上去

这样像 `/system/user` 这种地址刷新时，就不会先掉到 404。

#### 第二层：路由守卫兜底恢复

如果某些情况下浏览器已经先命中了静态 404，守卫里还会继续检测：

1. 当前是不是 `/404`
2. 有没有 `redirectedFrom`
3. 当前动态路由是否还没恢复

如果满足条件，就先恢复动态路由，再跳回原始地址。

1. `src/layout/components/LayoutBreadcrumb.vue`

它的核心思路很简单：

1. 读取当前路由的 `route.matched`
2. 过滤掉没有 title、hidden、breadcrumb=false 的路由记录
3. 把剩余记录渲染成面包屑
4. 非最后一项支持点击跳转

这样做的好处是：

1. 不需要单独维护一份面包屑配置
2. 路由变，面包屑就自动变
3. 结构简单，特别适合学习型项目

### 19.6 标签页 store 是怎么设计的

这一步重点增强了 `tabs store`。

当前文件：

1. `src/store/modules/tabs.js`
2. `src/hooks/useTagsView.js`

当前 store 主要负责：

1. 保存已访问标签页列表 `visitedViewList`
2. 保存需要缓存的页面名列表 `cachedViewNameList`
3. 新增标签页
4. 删除单个标签页
5. 关闭其他标签页
6. 关闭全部标签页
7. 查找当前标签页
8. 刷新后恢复标签页

### 19.7 为什么首页标签能固定

这一步新增了一个非常实用的思路：

1. 从静态路由里自动收集 `meta.affix = true` 的路由
2. 把它们转成固定标签页列表
3. 标签页初始化、恢复、关闭全部时，都先把这些固定标签保留下来

当前首页就是固定标签页，所以：

1. 首页不会被右上角关闭按钮误删
2. “关闭全部”后至少会保留首页
3. 刷新页面后，首页标签仍然存在

### 19.8 标签页组件现在支持哪些交互

当前 `src/layout/components/LayoutTagsView.vue` 已支持：

1. 新开标签
   通过监听路由变化自动加入标签列表。

2. 切换标签
   点击标签即可跳转到对应页面。

3. 关闭当前
   点击当前标签的关闭按钮，或通过右侧操作菜单关闭当前页。

4. 关闭其他
   保留当前标签和固定标签，其它标签全部关闭。

5. 关闭全部
   清空所有非固定标签，并跳回首页。

6. 固定首页标签
   首页不会显示关闭按钮。

### 19.9 为什么标签页刷新后还能恢复

标签页恢复依赖的是本地持久化：

1. 每次标签页变化后，都写入 `localStorage`
2. 项目启动时由 `tabs store` 统一恢复
3. 恢复时还会再次合并固定标签，避免首页丢失

所以即使刷新页面，也能保留之前打开过的标签。

### 19.10 这一步为什么没有做得太复杂

很多后台项目会把标签页做得非常重，例如：

1. 右键菜单
2. 拖拽排序
3. 多层缓存控制
4. 左右滚动定位
5. 标签页持久化版本迁移

这些能力不是不能做，但对当前这个学习型模板来说太早了。

所以这一版刻意只保留最核心、最常见、最容易理解的能力：

1. 菜单结构清楚
2. 面包屑自动生成
3. 标签页交互完整
4. 刷新恢复稳定

## 20. 第 13 步：多布局模式实现

### 20.1 这一步的目标

这一步要把后台的 Layout 从“只有一种左侧布局”，升级成“一个统一容器可以调度多种布局模式”。

当前需要支持三种布局：

1. 左侧布局
2. 顶部布局
3. 混合布局

同时还要保证：

1. 布局切换状态保存在 store 中
2. 页面刷新后布局模式仍然能恢复
3. 菜单、顶部栏、标签页、内容区都能继续复用
4. 结构不要写得太绕，尽量让初学者能看懂

### 20.2 这一步为什么要这样做

很多后台项目一开始只有左侧布局，后面一旦要加顶部布局或混合布局，就容易把整个 Layout 推翻重写。

为了避免这个问题，这一步采用了一个更稳的思路：

1. 先把 Layout 根组件变成“统一调度器”
2. 再把三种布局分别拆成独立模式组件
3. 菜单、Logo、Header、标签页、内容区继续作为公共组件复用

这样做的好处是：

1. 切换布局时只是切换“组合方式”
2. 菜单、标签页、内容区逻辑不用重复写三遍
3. 后续就算要继续加第四种布局，也只需要新增一个模式组件

### 20.3 这一步新增和调整了哪些目录

当前 `layout` 目录结构变成了这样：

```text
src/layout/
├─ index.vue
├─ modes/
│  ├─ LeftLayout.vue
│  ├─ TopLayout.vue
│  └─ MixLayout.vue
└─ components/
   ├─ LayoutBreadcrumb.vue
   ├─ LayoutHeader.vue
   ├─ LayoutLogo.vue
   ├─ LayoutMain.vue
   ├─ LayoutMenu.vue
   ├─ LayoutMenuItem.vue
   ├─ LayoutSidebar.vue
   ├─ LayoutTagsView.vue
   └─ LayoutTopNav.vue
```

### 20.4 每个部分现在分别负责什么

#### 1. `src/layout/index.vue`

它现在不再直接写死某一种布局，而是统一决定：

1. 当前应该渲染左侧布局、顶部布局还是混合布局
2. 需要把哪些菜单数据传给对应的布局模式组件
3. 当前是不是移动端
4. 移动端侧边栏是否打开

也就是说，`index.vue` 现在像一个“布局调度中心”。

#### 2. `src/layout/modes/*`

这里放三种真正的布局模式：

1. `LeftLayout.vue`
   最常见的后台左侧菜单布局。

2. `TopLayout.vue`
   顶部横向菜单布局，适合一级菜单较少的系统。

3. `MixLayout.vue`
   一级菜单放顶部，二级菜单放左侧，这是很多中后台项目常见的“混合布局”。

#### 3. `src/layout/components/*`

这里继续保留所有公共组件：

1. `LayoutHeader.vue`
   统一头部组件，现在支持布局切换、插槽扩展。

2. `LayoutSidebar.vue`
   统一侧边栏组件，支持是否显示 Logo。

3. `LayoutMenu.vue`
   统一菜单组件，既能用于左侧竖向菜单，也能用于顶部横向菜单。

4. `LayoutTopNav.vue`
   专门给混合布局准备的一级菜单导航。

5. `LayoutLogo.vue`
   可复用于侧边栏和顶部栏。

6. `LayoutTagsView.vue`
   统一标签页区域。

7. `LayoutMain.vue`
   统一内容区路由出口。

### 20.5 根布局是怎么统一调度不同模式的

这一步的关键点之一，是把 `src/layout/index.vue` 改成下面这种思路：

1. 从 `app store` 中拿到当前 `layoutMode`
2. 根据 `layoutMode` 计算出当前应该渲染哪个布局组件
3. 把菜单、移动端状态、折叠状态等公共数据传进去

这样就实现了：

1. 一个入口
2. 三种布局模式
3. 公共状态统一管理

这比在三个页面里各自处理状态要清晰得多。

### 20.6 布局切换状态为什么放到 store 里

布局模式本质上属于“应用级状态”。

它不是某个页面自己的临时状态，而是整个后台壳子的表现方式，所以更适合放进 `app store`。

当前相关文件：

1. `src/constants/layout.js`
2. `src/store/modules/app.js`
3. `src/hooks/useLayout.js`

现在的处理方式是：

1. `layoutMode` 保存当前布局模式
2. 调用 `setLayoutMode` 切换布局
3. 切换后自动持久化到本地缓存
4. 项目启动时再从缓存里恢复

所以即使刷新页面，当前布局模式也不会丢。

### 20.7 为什么要单独新增 `useLayoutMenu`

这一步新增了：

1. `src/hooks/useLayoutMenu.js`

它的作用是把“和布局有关的菜单派生逻辑”集中起来，例如：

1. 顶部布局要用完整一级菜单
2. 混合布局顶部要用一级菜单
3. 混合布局左侧要根据当前激活的一级菜单，决定显示哪一组二级菜单

如果这些逻辑都写在 `Layout.vue` 或某个布局组件里，代码会越来越绕。

所以这里单独抽成一个 hook，后续维护会更轻松。

### 20.8 三种布局分别是怎么复用公共组件的

#### 左侧布局

左侧布局最直接：

1. 左侧放 `LayoutSidebar`
2. 右侧依次放 `LayoutHeader`
3. 下面是 `LayoutTagsView`
4. 最后是 `LayoutMain`

这也是后台项目里最基础、最常见的布局。

#### 顶部布局

顶部布局没有桌面端左侧栏，所以变成：

1. 顶部 `LayoutHeader`
2. Header 左侧放 `LayoutLogo`
3. Header 中间放横向 `LayoutMenu`
4. 下方继续复用标签页和内容区

移动端下，顶部布局仍然会回退到抽屉式侧边栏，这样交互更统一。

#### 混合布局

混合布局的思路是：

1. Header 左侧放 `LayoutLogo`
2. Header 中间放 `LayoutTopNav`，只负责一级菜单
3. 左侧 Sidebar 只显示当前一级菜单对应的二级菜单
4. 下方继续复用标签页和内容区

这也是为什么混合布局会比普通顶部布局多一个 `LayoutTopNav.vue` 组件。

### 20.9 为什么头部组件要加插槽

这一步对 `LayoutHeader.vue` 做了一个很重要的升级：

1. 增加了 `brand` 插槽
2. 增加了 `center` 插槽
3. 增加了布局切换下拉菜单

这样做之后：

1. 左侧布局可以只显示折叠按钮、标题、面包屑
2. 顶部布局可以把 Logo 和横向菜单放进 Header
3. 混合布局可以把 Logo 和一级导航放进 Header

也就是说，Header 变成了“可组合”的公共头部，而不是只能服务某一种布局。

### 20.10 为什么菜单组件也要复用

这一步没有为了顶部布局再单独重写一套菜单，而是对 `LayoutMenu.vue` 做了增强：

1. 支持 `vertical` 模式
2. 支持 `horizontal` 模式
3. 支持通过 `theme` 区分侧边栏菜单和顶部菜单样式

这样做的好处是：

1. 菜单数据结构不用变
2. 菜单递归渲染逻辑不用重复写
3. 顶部布局和左侧布局仍然共享同一份菜单基础能力

### 20.11 混合布局的菜单为什么要分“顶部一级菜单”和“左侧二级菜单”

混合布局和普通布局最大的区别，就是菜单需要按层级拆开显示。

这一步当前采用的策略是：

1. 顶部导航只显示一级菜单
2. 左侧侧边栏只显示当前激活一级菜单对应的子菜单
3. 如果一级菜单没有 children，就左侧退回显示它自己，避免左侧空白

这个逻辑已经统一放在 `useLayoutMenu` 中了。

### 20.12 移动端为什么还保留抽屉式侧边栏

虽然桌面端有三种布局模式，但移动端上不适合继续坚持复杂布局差异。

所以当前策略是：

1. 小屏幕时，菜单统一回退为抽屉式侧边栏
2. 页面切换后自动关闭移动端菜单
3. 桌面端才真正区分左侧、顶部、混合三种展示方式

这样做的好处是：

1. 移动端交互更一致
2. 代码复杂度更低
3. 对初学者更容易理解

完成这一步后，项目已经从“单一后台壳子”升级成了“可切换的多布局后台壳子”。

当前已经具备：

1. 左侧布局
2. 顶部布局
3. 混合布局
4. 统一 Layout 调度入口
5. 布局模式 store 持久化
6. 菜单、Header、标签页、内容区复用
7. 移动端抽屉式菜单兼容

## 21. 第 14 步：Dashboard 首页

### 21.1 开始这一步之前，先修一个面包屑警告

这一步开始前，项目里先修了一个控制台警告：

```text
[Vue warn]: Duplicate keys found during update: "/system/user"
```

问题出在 `LayoutBreadcrumb.vue`：

1. 面包屑原来把 `redirect || path` 当成 `key`
2. 但 `/system` 这个父级路由的 `redirect` 正好也是 `/system/user`
3. 子路由自己的 `path` 也是 `/system/user`
4. 所以父子两项在更新时就撞 key 了

这次修复的思路是：

1. 单独生成一个 `key` 字段
2. 优先用路由 `name`
3. 没有 `name` 时再退回到 `path + index`
4. 跳转路径仍然保留 `redirect || path`

这样既不影响面包屑点击跳转，也能彻底消掉重复 key 警告。

### 21.2 这一步的目标

这一阶段要把首页从“演示页面”升级成真正像后台工作台的首页。

首页至少要包含：

1. 欢迎信息
2. 统计卡片
3. 快捷入口
4. 最近操作
5. 系统公告

同时还要做到：

1. `admin` 和 `editor` 看到的内容略有差异
2. 首页结构接近真实后台项目
3. 数据先走 mock
4. 后续很容易切到真实后端接口

### 21.3 为什么首页要这样设计

后台首页的作用不是“展示花哨效果”，而是帮助用户一进入系统就快速看到：

1. 我是谁
2. 当前最重要的事情是什么
3. 我最常做的入口在哪
4. 最近发生了什么
5. 有没有必须注意的系统通知

所以这一版首页刻意采用比较常见、也比较稳的工作台结构：

1. 顶部欢迎卡片先告诉用户当前角色和今日重点
2. 统计卡片让用户一眼看到关键数字
3. 快捷入口减少重复找菜单的成本
4. 最近操作帮助用户快速回忆上下文
5. 系统公告把重要通知提前暴露在首页

### 21.5 每个文件分别负责什么

#### 1. `src/api/dashboard.js`

首页接口统一从这里出去。

当前开发阶段：

1. 如果开启 mock，就走本地 mock 数据
2. 如果关闭 mock，就走真实接口 `/dashboard/home`

这样首页页面本身不需要知道“当前到底用的是 mock 还是真接口”。

#### 2. `src/mock/data/dashboard.js`

这里专门保存首页 mock 数据内容。

当前按角色拆成两套：

1. `adminDashboardData`
2. `editorDashboardData`

这样做的好处是：

1. 首页结构保持统一
2. 角色不同，内容自然不同
3. 后续如果再加第三种角色，也很方便扩展

#### 3. `src/mock/modules/dashboard.js`

这个文件负责：

1. 根据 token 找到当前账号
2. 判断当前账号属于哪个角色
3. 返回对应角色的首页数据

也就是说，真正“按角色返回首页内容”的逻辑，是放在这里完成的。

#### 4. `src/views/dashboard/index.vue`

页面层只负责：

1. 调用首页接口
2. 保存当前首页数据
3. 处理快捷入口点击行为
4. 组合各个子组件

这样页面本身不会被太多展示细节拖乱。

#### 5. `src/views/dashboard/components/*`

这些组件负责首页每一个独立板块：

1. `DashboardWelcomeCard.vue`
   欢迎区和顶部概览信息。

2. `DashboardStatCard.vue`
   单个统计卡片。

3. `DashboardShortcutPanel.vue`

4. `DashboardRecentActionPanel.vue`
   最近操作时间线。

5. `DashboardNoticePanel.vue`
   系统公告列表。

### 21.6 为什么首页要拆组件，而不是全写在一个文件里

如果首页所有内容都写在 `index.vue` 里，短期看起来省事，但很快就会出现这些问题：

1. 模板太长，不容易看
2. 样式全部堆在一起，不好维护
3. 后续某一块要单独改版时，改动范围太大
4. 页面层会同时承担“请求数据”和“负责排版”两类职责

所以这一步采用比较稳的拆分方式：

1. 页面层负责“拿数据和调度组件”
2. 子组件负责“展示某一块内容”

这对于学习者来说也更友好，因为每个文件的职责非常直观。

### 21.7 首页数据结构是怎么设计的

当前首页数据结构大致分成五块：

1. `welcome`
   欢迎文案、角色标签、最近登录时间、重点提醒、顶部概览数字。

2. `statCardList`
   统计卡片数组，每一项包含标题、数值、单位、趋势、说明和强调色。

3. `shortcutList`
   快捷入口数组，支持两类行为：
   `route` 跳已有页面，`action` 模拟真实业务动作。

4. `recentActionList`
   最近操作记录列表，用来渲染时间线。

5. `noticeList`
   系统公告列表，用来展示首页右侧通知区域。

这种结构的好处是：

1. 语义清楚
2. 页面读取简单
3. mock 和真实接口都容易对齐

### 21.8 admin 和 editor 为什么会看到不同内容

这一步不是简单地在页面里写 `if admin else editor`，而是让接口返回不同数据。

当前 mock 的处理方式是：

1. 登录后已经有 token
2. 首页接口通过 token 找到当前账号
3. 再根据角色返回 `admin` 或 `editor` 对应的首页数据

所以：

1. `admin` 首页更偏系统稳定性、权限巡检、账号管理
2. `editor` 首页更偏稿件发布、内容排期、评论反馈

这样比在页面里硬编码分支更接近真实项目。

### 21.9 快捷入口为什么同时支持 route 和 action

后台首页里的快捷入口不一定都已经有完整页面。

例如：

1. 有些入口已经有对应路由，可以直接跳转
2. 有些入口目前只是未来能力的预留，还没有真正页面

所以这里做了一个很实用的设计：

1. `type = route`
   直接跳转到现有页面

2. `type = action`
   先用消息提示模拟真实业务动作

这样既能保证首页结构完整，也不会为了一个入口临时造出很多还没准备好的页面。

### 22.3 为什么后台项目建议统一页面模板

1. 页面长得不统一
2. 查询区、表格区、按钮区布局风格不一致
3. 分页写法、返回交互、头部结构各不相同
4. 新同学接手后，需要每个页面重新理解一遍

所以后台项目通常都会尽量统一页面模板。

1. 结构统一，页面更像一个系统，而不是很多零散页面拼起来

## 24. 第 16 步：后台管理项目的移动端兼容方案

### 24.1 这一步的目标

这一版不是要把项目做成纯移动端后台，而是要保证在常见小屏设备下：

1. 布局不会直接挤坏
2. 菜单还能正常打开和关闭
3. 头部区域不会乱堆到一起
4. 列表和表单还能继续使用

### 24.2 这一步采用的总体策略

这一版移动端适配采用的是“状态判断 + 结构降级 + 局部横向滚动”的简单方案。

核心思路如下：

1. 用 pp store 统一维护 deviceType、isMobile、mobileSidebarVisible
2. 布局层在小屏下自动把侧边栏切换成抽屉式滑出菜单
3. 头部区域在小屏下允许收缩和换行，而不是强行压缩所有内容
4. 表格不强行挤成窄列，而是允许横向滚动
5. 表单通过通用模板统一做小屏兼容，减少每个业务页重复写适配样式

## 26. 补充修复记录：侧边栏父菜单展开状态受控化

### 26.1 问题现象

原来的侧边栏父菜单展开状态存在两个体验问题：

1. 用户手动展开了“系统管理”后，如果当前停留在“首页”并刷新页面，父菜单会自动收起
2. 用户先进入“用户管理”等子页面，再切换回“首页”，父菜单也可能自动收起

这会导致已经展开过的父菜单状态丢失，不符合后台系统常见的使用习惯。

### 26.2 问题根因

根因并不在权限路由，而是在侧边栏菜单的展开状态没有被单独保存。

原来的实现里，菜单更多依赖 el-menu 内部状态：

1. 当前激活菜单和父菜单展开状态没有彻底拆开
2. 页面切换、刷新、重渲染后，父菜单展开状态缺少稳定恢复来源
3. 没有把用户手动展开过的父菜单持久化到 store 或本地缓存中

### 26.3 这次修复的核心思路

这次改成了“受控展开状态”方案：

1. 在 pp store 中新增 openedMenuPathList
2. 用它专门保存父菜单的展开路径
3. 展开状态写入本地缓存，刷新后恢复
4. LayoutMenu 不再把“当前激活菜单”和“当前展开菜单”混用
5. 页面切换后，会继续按照 store 中保存的展开状态重新同步菜单

### 26.4 具体改动位置

#### 1. app store

src/store/modules/app.js

新增了：

1.  openedMenuPathList
2.  setOpenedMenuPathList
3.  ddOpenedMenuPath
4.  emoveOpenedMenuPath
5.           oggleOpenedMenuPath

并且把它纳入了应用级持久化状态中。

#### 2. useLayout hook

src/hooks/useLayout.js

把侧边栏展开状态相关字段和方法统一暴露出来，方便布局组件直接复用。

#### 3. LayoutMenu 组件

src/layout/components/LayoutMenu.vue

这一步是最关键的：

1. 去掉了依赖内部默认行为的 unique-opened
2. 增加 isibleOpenedMenuPathList 作为当前菜单树下真正需要展开的父菜单列表
3. 通过 handleMenuOpen / handleMenuClose 把用户手动展开和收起操作同步到 store
4. 通过 syncMenuOpenedState 在页面切换、菜单重渲染、折叠恢复后重新同步展开状态

### 26.5 修复后的行为规则

现在侧边栏已经满足下面这些规则：

1. 父菜单展开状态单独保存，不再依赖当前路由临时推断
2. 用户手动展开过的父菜单，会持续保持展开
3. 页面切换时，不会自动清空展开状态
4. 刷新页面后，会恢复上一次展开状态
5. 只有用户再次点击当前父菜单时，才会收起
6. 点击子菜单时，不会影响父菜单的展开状态
7. 当前激活菜单和父菜单展开状态已经分开管理

### 26.6 这次修复解决了什么

做完这次修复后，侧边栏交互更接近真实后台系统：

1. 菜单更稳定
2. 用户上下文不会轻易丢失
3. 展开状态、激活状态、权限菜单三者职责更清晰
4. 后续如果继续扩展更多多级菜单，也可以继续沿用这套受控展开方案

### 27.3 为什么要先做一个通用状态组件

很多初学者在做后台项目时，容易把异常页、空状态、加载失败分开写，最后会出现几个问题：

1. 每个页面风格都不一样
2. 重复代码越来越多
3. 修改一次视觉风格要改很多文件
4. 页面级异常和区块级异常没有统一语言

所以这里先做一个 `AppStatus`，让它承担“统一异常表现层”的角色。
后续你无论是做：

1. 列表页无数据
2. 详情页加载失败
3. 页面无权限
4. 服务异常页

都可以优先从这个组件复用，而不是再重新拼一套结构。

### 27.4 当前异常处理方案是怎么设计的

#### 1. 页面级异常

当前已经有：

1. `/403`：无权限页
2. `/404`：页面不存在页
3. `/500`：服务异常页

其中：

1. `403` 放在 `Layout` 里，用户在系统内部被拦到无权限页时，仍然保留后台壳子
2. `404` 保持独立，更适合作为全局兜底页
3. `500` 当前作为“按需启用”的服务异常页，不会让所有请求失败都直接跳进去

#### 2. 请求异常统一提示

axios 请求层现在已经统一处理：

1. `401`：清理登录状态并跳转登录页
2. `403`：统一跳转到 `/403`
3. `500/502/503/504`：统一提示错误消息，并支持按需进入 `/500`
4. 其他错误：统一弹出消息提示

#### 3. 为什么 500 页不是默认所有异常都跳

这里故意没有把“所有请求失败”都跳到 `500`，因为后台里很多错误其实只是一个局部区块加载失败。
如果一出错就整页跳走，会让交互体验太重。

所以当前采用“默认消息提示 + 可选进入 500 页”的方案：

1. 默认只弹提示
2. 如果某个页面希望出现整页异常效果，可以在请求时传 `showErrorPage: true`

示例：

```js
getRequest(
  '/report/overview',
  {},
  {
    showErrorPage: true
  }
)
```

### 27.5 页面无数据和加载失败如何统一风格

这一步除了错误页，还把 Dashboard 的区块状态统一了：

1. 快捷入口为空时，不再直接用默认 `el-empty`
2. 最近操作为空时，也统一走 `AppStatus`
3. 系统公告为空时，也统一走 `AppStatus`
4. 首页整体加载失败时，显示统一的“加载失败”状态块，并提供“重新加载”按钮

这就是一个很典型的后台模板思路：

1. 页面级异常走异常页
2. 区块级异常走通用状态组件
3. 请求层统一兜底错误提示

### 27.6 给后续页面的复用建议

后面做列表页、详情页、表单页时，建议优先按下面方式复用：

1. 无权限：跳转 `/403`
2. 页面不存在：跳转 `/404`
3. 页面级服务异常：使用 `/500` 或 `AppStatus type="error"`
4. 区块无数据：使用 `AppStatus type="empty" mode="section"`
5. 页面加载失败：使用 `AppStatus type="error"`

### 28.4 路由为什么要加语言前缀

这一步最核心的结构升级，是把语言直接放进路由路径里，例如：

1. `/zh-CN/login`
2. `/zh-CN/dashboard`
3. `/en-GB/system/user`

这样做的好处很明显：

1. 当前页面语言是“可见”的，不再只是藏在内存里
2. 页面刷新后，语言状态天然可恢复
3. 复制 URL 给别人时，对方打开后能直接进入同语言页面
4. 动态路由、权限路由、404 路由都能和语言状态天然绑定

当前实现方式是：

1. 根路径 `/` 自动重定向到当前偏好语言首页
2. 无语言前缀的地址会自动补全当前语言前缀
3. 登录页、403、404、500、Dashboard、System 等路由全部变为带 `:locale` 的结构

### 28.5 为什么菜单、标签页、面包屑不用直接保存完整语言路径

如果把菜单状态直接存成：

1. `/zh-CN/system`
2. `/en-GB/system`

那么一旦切语言：

1. 菜单展开状态会丢
2. 顶部菜单激活状态会乱
3. 标签页会重复生成两套

所以这一步改成了：

1. 菜单、标签页、展开状态统一保存“基准路径”
   - `/system`
   - `/system/user`
   - `/dashboard`
2. 真正跳转时，再用 `resolveLocalePath()` 补上当前语言前缀

这就是为什么这一轮专门在 `router/helper.js`、`LayoutMenu.vue`、`LayoutBreadcrumb.vue`、`LayoutTagsView.vue`、`tabs store` 里都做了路径归一化处理。

## 补充修复：vite.config.js 编码与 Element Plus 样式导入

### 现象

某次修改后，`vite.config.js` 出现了两个问题：

1. 文件内容中的中文注释显示为乱码
2. 编辑器或运行环境会提示配置文件异常，影响继续维护

同时，项目里还存在一个潜在问题：Element Plus 的自动导入虽然已经接入，但如果没有显式要求“按需导入对应样式”，那么像 `ElMessage`、`ElMessageBox`、`ElNotification` 这类服务型组件，可能会出现样式不完整的情况。

### 这次修复做了什么

这次没有继续在原有乱码内容上打补丁，而是直接把 `vite.config.js` 整份重写为正常的 UTF-8 内容，并保留原来的配置结构。

同时，把下面两处 `ElementPlusResolver()` 都改成了：

```js
ElementPlusResolver({
  importStyle: 'css'
})
```

分别用于：

1. `AutoImport` 中的服务型 API 自动导入
2. `Components` 中的组件自动导入

### 为什么这样修

原因有两个：

1. 乱码文件继续局部修改，后面很容易再出编码问题，不如直接整体修正
2. `importStyle: 'css'` 能让 Element Plus 在按需导入组件或服务时，把对应样式文件一起带上，比较适合当前这个 `Vue 3 + Vite + JavaScript + Less` 的学习型后台模板项目

### 修复结果

1. `vite.config.js` 已恢复为可正常维护的版本
2. 配置文件语法检查通过
3. `pnpm lint` 通过
4. Element Plus 样式按需导入链路更完整

## 29. 布局与标签页补充修复

### 29.1 本轮要解决的问题

1. 界面设置弹窗在移动端没有做自适应，小屏下使用体验不够友好。
2. PC 端左侧菜单栏没有固定住，右侧内容滚动时左侧也会跟着一起滚。
3. 新增的 `VueUse Demo` 页面点击后没有生成对应的标签页，也没有接入缓存。

### 29.2 这次排查出来的根因

1. 头部设置面板一直使用 `el-popover`，这种形态更适合桌面端，不适合移动端做大面积设置面板。
2. 当前布局壳层虽然用了 `min-height: 100vh`，但没有把右侧主内容区真正做成独立滚动容器，所以实际滚动还是落在页面根节点上。
3. `tabs store` 在生成标签页时，直接读取了合并后的 `route.meta`。由于布局父路由本身配置了 `hidden: true`，子页面也被误判成隐藏页，于是像 `VueUse Demo` 这样的子页面不会进入 tagsView。
4. `LayoutMain.vue` 之前还没有接入 `KeepAlive`，即使某个路由写了 `meta.keepAlive`，页面实例也不会真正被缓存下来。

### 29.3 这次具体改了什么

1. 新增 `src/layout/components/LayoutSettingsPanel.vue`，把界面设置面板主体内容拆成独立组件，避免头部组件继续膨胀。
2. `src/layout/components/LayoutHeader.vue` 改成“桌面端 `el-popover` + 移动端 `el-drawer`”双形态实现：
   - PC 端继续保留轻量弹出层交互
   - 移动端改成抽屉，宽度可以跟随视口，更适合连续设置布局、主题、语言
3. `src/layout/index.vue`、`src/layout/components/LayoutSidebar.vue`、`src/layout/components/LayoutMain.vue` 重新梳理了布局高度和滚动职责：
   - 整个布局壳层固定为视口高度
   - 右侧主内容区自己滚动
   - 左侧菜单栏不再跟着页面正文一起滚动
4. `src/store/modules/tabs.js` 调整为优先读取当前“叶子路由记录”的 `meta` 和 `name`，避免父级布局路由的 `hidden` 配置误伤子页面标签页生成逻辑。
5. `src/router/modules/static.js` 给 `VueUseDemo` 路由补上了 `keepAlive: true`。
6. `src/layout/components/LayoutMain.vue` 正式接入 `KeepAlive`，并结合 `tabsStore.cachedViewNameList` 来控制缓存页面名单。
7. 由于主内容区现在改成了内部滚动容器，所以顺手补上了“切换路由后把主内容区滚动条重置到顶部”的逻辑，保持后台系统常见体验。

### 29.5 这一轮修改后的结果

1. 移动端的界面设置入口不再挤在小弹层里，而是切换成更适合手机的抽屉面板。
2. PC 端布局滚动职责更清晰，左侧菜单栏保持固定，只滚右侧内容区域。
